## 用語
ここのセキュリティチェックは「コントロール」
コントロールの集合体が「セキュリティ標準」
チェック結果に応じて「セキュリティスコア」が算出
要件を満たしていれば「成功（合格）」
コントロールごとに「重大ど」が決まっている

## コントロールの成功・失敗
コントロールに重要度がついている
あるコントロールに対して対象リソースが全て成功の場合、成功
一個でも失敗の場合は、失敗

## 状況に応じたカスタマイズ
* コントロールの**無効化**
-> サービス不使用など
* コントロールでチェックするリソースを**抑制**
-> 開発環境でのバックアップはコストと不要だな

## AWS Configのコスト対策
* AWS Configuのコストが高い
-> Security Hub CSPMはAWS Configを有効かする必要がある
-> AWS Configの設定を取得して評価している
-> Configの記録頻度を調整してコスト対策
-> 連続 -> 日次 -> 除外

# 大道さん
クラウド移管でDMSを利用した悪い思い出

## DMSとは
Database Migration Service
オンプレミスや他のクラウドからAWSへデータを移行するサービス
* 同種移行・異種以降が可能
* リアルタイムの複製やレプリケーションが可能
* データをフィルタリングもある

## DMSに泣いた話
フルロード24時

フルロード -> 移行のみ
フルロード＋レプリケーション -> 移行 + 複製 継続的に
レプリケーション -> 継続的に複製のみ

進まない
-> T3系インスタンスを使用していた -> バーストありのモデル
-> T3系以外に変更 -> 初期設定がT3

メトリクス見ておくと良い
-> BurstBalance
-> CPUUtilization
-> FreeableMemory

-> フルロードは進んだ！
-> 進捗7% -> 9時間で -> 朝6時に止めた

遅い原因
-> データに大量のレコードがある
* タスク分割
-> カラム数が多いテーブルがある
* タスク分割
-> スペック不足
* レプリケーションインスタンスの調整

-> 1時間で44%進むようになった！

## レプリケーション遅延
遅延が4.5時間ほど発生
-> 旧環境に問題がありそう
-> 原因わからず...

AWS切り替えまで6時間猶予がある
-> いける
-> 移行スケジュール再現性を重視
-> 移行テストを4回実施
-> レプリケーション遅延が22時〜23時で解消

## まとめ
フルロードはインスタンスタイプとタスク設計
レプリケーション遅延は運用の再現性を優先
